## 1. Introduction to Angular Services: Enhancing Code Organization and Reusability

Welcome to this comprehensive lesson on Angular Services and Dependency Injection. In modern web development, especially with frameworks like Angular, maintaining a clean, organized, and scalable codebase is paramount. As applications grow in complexity, the need to share logic, data, and functionality across different parts of the application becomes critical. This is precisely where Angular Services shine. They act as the backbone for reusable business logic, data fetching, and state management, ensuring that your components remain focused on presentation rather than complex operations. This lesson will delve deep into what services are, how to create and utilize them effectively, and the powerful mechanism of Dependency Injection that makes them so seamless to integrate. By the end of this session, you will understand how to leverage services to build more robust, maintainable, and efficient Angular applications, directly contributing to the module's learning objectives: 'Understand the role of services in Angular.' and 'Create and inject services.' This knowledge is foundational for building sophisticated applications, particularly when interacting with backend APIs using the HTTP Client, a topic we will explore further.

**Real-world Relevance:** Imagine building an e-commerce application. You'll need to fetch product lists, manage user authentication, handle shopping cart operations, and potentially interact with various backend APIs. Instead of duplicating this logic in every component that needs it, services allow you to encapsulate these functionalities in one place, making them accessible and manageable from anywhere in your application. This not only reduces code duplication but also simplifies testing and maintenance. For instance, a `ProductService` could be responsible for all interactions with the product catalog API, while an `AuthService` handles user login and logout. This separation of concerns is a cornerstone of good software design and is heavily promoted by Angular's architecture.

## 2. Understanding the Core Concept: What are Angular Services?

At its heart, an **Angular Service** is a class that encapsulates a specific piece of functionality. This functionality can range from fetching data from an API, performing complex calculations, managing application state, to interacting with browser APIs like local storage. Services are designed to be reusable and shareable across different components, directives, and other services within your Angular application. They promote the principle of **Separation of Concerns** , allowing you to keep your components lean and focused on the user interface, while delegating complex logic or data management tasks to services.

**Why are Services Important?**

  * **Reusability:** Write logic once and use it in multiple places. This drastically reduces code duplication and the potential for errors. If you need to update how data is fetched, you only need to change it in one service file.
  * **Maintainability:** By centralizing functionality, services make your codebase easier to understand, debug, and maintain. When a bug related to data fetching occurs, you know exactly where to look – in the relevant data service.
  * **Testability:** Services are typically plain TypeScript classes, making them easy to unit test in isolation. This is crucial for building reliable applications. You can mock services during component testing to simulate different scenarios without needing a live backend.
  * **Organization:** Services help structure your application logically. They group related functionalities, making it easier to navigate and manage your project as it grows.
  * **Decoupling:** Services decouple components from specific implementation details, such as how data is fetched or how certain calculations are performed. This makes your components more flexible and adaptable to changes.



Think of services as specialized workers in your application. A `UserService` might be responsible for all user-related operations, a `LoggerService` for handling application logs, and a `HttpClientService` (which we'll build upon later) for making HTTP requests to your ASP.NET Core backend. They are not tied to a specific view or component and can be instantiated and managed by Angular's injector system.

The decision to use a service is often made when you identify a piece of logic or data that is:

  * **Needed by multiple components:** If more than one component requires the same functionality, it's a prime candidate for a service.
  * **Complex:** Logic that involves intricate calculations, data transformations, or interactions with external systems is best placed in a service.
  * **Stateful:** Services can maintain state across the application, which is useful for things like user authentication status or shared data models.
  * **Independent of the UI:** Functionality that doesn't directly relate to rendering UI elements or handling user interactions should be in a service.



In essence, services are the building blocks for creating robust, scalable, and maintainable Angular applications by promoting modularity and reusability.

## 3. Generating Angular Services with the Command Line Interface

Angular's powerful Command Line Interface (CLI) simplifies the process of creating new components, modules, services, and other application building blocks. Generating a service using the CLI is a straightforward process that sets up the necessary file structure and boilerplate code for you. This ensures consistency and adherence to Angular's best practices.

**The Command:**

To generate a new service, you'll use the `ng generate service` command, often aliased as `ng g service`. You specify the name of the service you want to create. For example, to create a service named `DataService`, you would run the following command in your terminal, within your Angular project's root directory:
    
    
    ng generate service data

Or, using the alias:
    
    
    ng g s data

**What the CLI Generates:**

When you run this command, the Angular CLI will typically create two files:

  1. `src/app/data.service.ts`: This is the main TypeScript file for your service. It will contain the basic structure of the service class.
  2. `src/app/data.service.spec.ts`: This file contains a basic unit test file for your service. Angular encourages testing from the outset, and the CLI provides a starting point for this.



**Example Output Structure:**

The generated `data.service.ts` file will look something like this:
    
    
    import { Injectable } from '@angular/core';
    
    @Injectable({
      providedIn: 'root'
    })
    export class DataService {
    
      constructor() { }
    
    }
    

The generated `data.service.spec.ts` file will contain a basic test suite:
    
    
    import { TestBed } from '@angular/core/testing';
    
    import { DataService } from './data.service';
    
    describe('DataService', () => {
      let service: DataService;
    
      beforeEach(() => {
        TestBed.configureTestingModule({});
        service = TestBed.inject(DataService);
      });
    
      it('should be created', () => {
        expect(service).toBeTruthy();
      });
    });
    

**Specifying a Path:**

You can also specify a path for your service. For instance, if you want to create a service within a specific feature folder, say `src/app/services/data`, you can do so:
    
    
    ng generate service services/data

This would create the service files within the `src/app/services/data` directory.

**The`--skip-tests` Flag:**

If you prefer to create the test file manually later or are not focusing on testing at this stage, you can use the `--skip-tests` flag:
    
    
    ng generate service data --skip-tests

This command streamlines the creation process by omitting the `.spec.ts` file. However, for comprehensive development, it's generally recommended to keep the test files.

Using the CLI for service generation is the standard and most efficient way to start building your reusable logic modules in Angular. It ensures that your project structure remains organized and that you are following established conventions.

## 4. The Essential `@Injectable()` Decorator: Enabling Dependency Injection

The `@Injectable()` decorator is a cornerstone of Angular's Dependency Injection (DI) system. It marks a class as a potential provider of services and allows Angular's injector to manage its instantiation and injection into other classes. Without this decorator, Angular's DI system wouldn't know how to create an instance of your service or how to inject it where needed.

**What is Dependency Injection?**

Dependency Injection is a design pattern where a class receives its dependencies (other objects it needs to function) from an external source rather than creating them itself. In Angular, the **Injector** is responsible for creating instances of services and providing them to components or other services that declare them as dependencies. This promotes loose coupling and makes code more modular and testable.

**The Role of `@Injectable()`:**

When you apply the `@Injectable()` decorator to a class, you are essentially telling Angular:

  * **This class can be injected as a dependency:** Other classes can request an instance of this service.
  * **This class might have its own dependencies:** If your service itself needs other services, the `@Injectable()` decorator helps Angular's DI system resolve those dependencies as well.



**Basic Usage:**

As seen in the CLI-generated service, the decorator is applied directly above the class definition:
    
    
    import { Injectable } from '@angular/core';
    
    @Injectable()
    export class MyService {
      // ... service logic ...
    }
    

**The `providedIn` Property:**

The `@Injectable()` decorator can also take an optional configuration object. The most common and recommended property is `providedIn`. This property specifies how and where the service should be provided (made available) in your application's dependency injection tree.

  * **`providedIn: 'root'`:** This is the most common and preferred way to provide a service. When you set `providedIn: 'root'`, Angular registers the service at the application's root level. This means a single instance of the service is created for the entire application, and it's available for injection anywhere. This is known as a **singleton** service. Using `'root'` also enables **tree-shaking** , meaning if the service is never actually used in the application, it can be removed from the final build, reducing bundle size.
  * **`providedIn: ModuleName`:** You can also provide a service at the module level. For example, `providedIn: MyFeatureModule` would make the service available only to components and services within `MyFeatureModule` and its children. This is less common than `'root'` but useful for feature-specific services.



**Example with`providedIn: 'root'`:**
    
    
    import { Injectable } from '@angular/core';
    
    @Injectable({
      providedIn: 'root' // This makes it a singleton service available app-wide
    })
    export class DataService {
      constructor() { }
    
      getMockData(): string[] {
        return ['Item 1', 'Item 2', 'Item 3'];
      }
    }
    

**When is `@Injectable()` strictly necessary?**

While it's good practice to always include `@Injectable()` on your services, it's technically only required if your service itself has dependencies that need to be injected into its constructor. If a service has no constructor dependencies, Angular can often infer its providers. However, explicitly using `@Injectable()`, especially with `providedIn: 'root'`, is the standard and recommended approach for all services.

In summary, the `@Injectable()` decorator is the key that unlocks Angular's powerful Dependency Injection system for your services, enabling them to be easily created, managed, and utilized throughout your application.

## 5. Registering Services: Making Them Available for Injection

For Angular's Dependency Injection system to provide an instance of a service, that service must be **registered** as a provider. This registration tells Angular how to create the service and where it should be available in the application's DI hierarchy. There are several ways to register providers, but the most common and recommended methods involve using the `@Injectable()` decorator itself or configuring providers within an Angular module.

**1\. Using`@Injectable({ providedIn: 'root' })` (Recommended):**

As discussed in the previous section, this is the most modern and efficient way to register a service. When you use `providedIn: 'root'`, Angular automatically registers the service at the application's root injector. This makes the service a singleton and enables tree-shaking. You don't need to explicitly list it in any module's `providers` array.
    
    
    // src/app/services/data.service.ts
    import { Injectable } from '@angular/core';
    
    @Injectable({
      providedIn: 'root'
    })
    export class DataService {
      constructor() { }
    
      getData(): string[] {
        return ['Apple', 'Banana', 'Cherry'];
      }
    }
    

This approach is clean, concise, and leverages Angular's optimizations effectively. For most application-wide services, this is the preferred method.

**2\. Registering Providers in an Angular Module:**

Before the `providedIn` property became the standard, services were typically registered in the `providers` array of an Angular module (either the root module, `AppModule`, or a feature module). This method is still valid and useful in certain scenarios, particularly when you need more control over the service's lifecycle or scope.

**a) Root Module Registration (`AppModule`):**

If you don't use `providedIn: 'root'`, you would register your service in the `providers` array of your `AppModule`. This also makes the service a singleton for the entire application.
    
    
    // src/app/app.module.ts
    import { NgModule } from '@angular/core';
    import { BrowserModule } from '@angular/platform-browser';
    import { AppComponent } from './app.component';
    import { DataService } from './services/data.service'; // Assuming DataService is in ./services/data.service
    
    @NgModule({
      declarations: [
        AppComponent
      ],
      imports: [
        BrowserModule
      ],
      providers: [
        DataService // Registering DataService here
      ],
      bootstrap: [AppComponent]
    })
    export class AppModule { }
    

In this case, the `DataService` class itself might not need the `providedIn: 'root'` property, or it could be `@Injectable()` without any arguments if it has no dependencies.

**b) Feature Module Registration:**

You can also register services within feature modules. This makes the service available only to components and other services within that specific module and its children. This is useful for encapsulating feature-specific logic and preventing potential naming conflicts.
    
    
    // src/app/features/feature.module.ts
    import { NgModule } from '@angular/core';
    import { CommonModule } from '@angular/common';
    import { FeatureComponent } from './feature.component';
    import { FeatureService } from './feature.service'; // A service specific to this feature
    
    @NgModule({
      declarations: [
        FeatureComponent
      ],
      imports: [
        CommonModule
      ],
      providers: [
        FeatureService // Registering FeatureService here
      ]
    })
    export class FeatureModule { }
    

When a service is provided in a feature module, Angular creates a separate injector for that module. If the same service is provided in multiple feature modules, each module will have its own instance of the service, unless it's also provided at the root level.

**3\. Providing Services at Component Level:**

While less common for typical services, you can also register a service provider directly within a component's decorator. This makes the service instance specific to that component and its children. Each instance of the component will get its own instance of the service.
    
    
    // src/app/components/my-component/my-component.component.ts
    import { Component } from '@angular/core';
    import { ComponentService } from './component.service';
    
    @Component({
      selector: 'app-my-component',
      templateUrl: './my-component.component.html',
      providers: [ComponentService] // Providing ComponentService here
    })
    export class MyComponentComponent {
      constructor(private componentService: ComponentService) { }
    }
    

This approach is generally used for services that manage state or behavior specific to a particular UI element and should not be shared globally or across different component instances.

**Choosing the Right Registration Method:**

  * **`providedIn: 'root'`:** For application-wide singletons. Most common and recommended.
  * **Module`providers`:** For feature-specific services or when you need to override a root-provided service within a specific module.
  * **Component`providers`:** For component-specific state or logic.



Understanding these registration strategies is key to effectively managing the scope and lifecycle of your services in Angular.

## 6. Injecting Services into Components: Accessing Reusable Logic

Once a service is registered as a provider, Angular's Dependency Injection system makes it incredibly easy to inject and use that service within your components, directives, or even other services. The primary mechanism for this is through the component's constructor.

**The Process of Injection:**

To inject a service into a component, you simply declare a parameter in the component's constructor with the type of the service you want to inject. Angular's injector will then look for a provider for that service type. If found, it will create an instance of the service (or retrieve an existing one if it's a singleton) and pass it to the constructor when the component is instantiated.

**Step-by-Step Injection:**

  1. **Ensure the Service is Provided:** First, make sure your service is registered as a provider. The easiest way is using `@Injectable({ providedIn: 'root' })`. If not, ensure it's listed in the `providers` array of an appropriate module.

  2. **Declare the Service in the Component Constructor:** In your component's TypeScript file, add a parameter to its constructor. The parameter's type should be the service class you want to inject.

  3. **Use an Access Modifier (e.g.,**`private`**,**`public`**,**`protected`**):** It's standard practice to use an access modifier (like `private`) for the constructor parameter. This automatically creates a class property with the same name and type, making the service instance accessible throughout the component's class.




**Example: Injecting**`DataService`**into a Component**

Let's assume we have our `DataService` set up with `providedIn: 'root'` and a method `getData()` that returns mock data.
    
    
    // src/app/services/data.service.ts
    import { Injectable } from '@angular/core';
    
    @Injectable({
      providedIn: 'root'
    })
    export class DataService {
      constructor() { }
    
      getData(): string[] {
        console.log('Fetching mock data from DataService...');
        return ['Item A', 'Item B', 'Item C', 'Item D'];
      }
    }
    

Now, let's inject this service into a component, say `MyListComponent`.
    
    
    // src/app/components/my-list/my-list.component.ts
    import { Component, OnInit } from '@angular/core';
    import { DataService } from '../../services/data.service'; // Adjust path as needed
    
    @Component({
      selector: 'app-my-list',
      templateUrl: './my-list.component.html',
      styleUrls: ['./my-list.component.css']
    })
    export class MyListComponent implements OnInit {
      items: string[] = [];
    
      // Injecting DataService via the constructor
      constructor(private dataService: DataService) { }
    
      ngOnInit(): void {
        // Use the injected service instance to call its methods
        this.items = this.dataService.getData();
      }
    }
    

And in the component's template:
    
    
    <!-- src/app/components/my-list/my-list.component.html -->
    
    <h2>My List Items</h2>
    
    <ul>
      <li *ngFor="let item of items">{{ item }}</li>
    </ul>

**Explanation:**

  * In the `MyListComponent` constructor, `private dataService: DataService` tells Angular: "I need an instance of `DataService`."

  * Angular's injector finds the provider for `DataService` (because it's provided in root).

  * It creates or retrieves the singleton instance of `DataService`.

  * It passes this instance to the constructor, and the `private` keyword automatically assigns it to the component's `dataService` property.

  * In `ngOnInit`, we can now call `this.dataService.getData()` to retrieve the data and populate the component's `items` array.




**Benefits of Constructor Injection:**

  * **Clear Dependencies:** The constructor clearly lists all the dependencies a component needs.

  * **Testability:** During unit testing, you can easily provide mock instances of services to the component's constructor, allowing you to test the component's logic in isolation.

  * **Readability:** It's a clean and standard way to manage dependencies.




This pattern of injecting services via the constructor is fundamental to building Angular applications and is used extensively when interacting with backend APIs via the HTTP Client, as the HTTP Client itself is typically injected as a service.

## 7. The Singleton Pattern for Services: Ensuring a Single Source of Truth

The **Singleton pattern** is a design pattern that ensures a class has only one instance and provides a global point of access to it. In the context of Angular services, this pattern is incredibly valuable for managing application-wide state, configurations, or shared resources. When a service is implemented as a singleton, all parts of your application that inject this service will receive the exact same instance. This is crucial for maintaining consistency and avoiding conflicting states.

**How Angular Achieves Singletons:**

Angular's Dependency Injection system, particularly when combined with the `@Injectable({ providedIn: 'root' })` decorator, automatically implements the singleton pattern for services. Here's how it works:

  1. **Root Injector:** When you use `providedIn: 'root'`, Angular registers the service with the application's root injector. The root injector is the top-level injector in the DI hierarchy and is responsible for creating and managing singleton instances of services.
  2. **Lazy Loading and Tree-Shaking:** A significant advantage of `providedIn: 'root'` is that it enables lazy loading and tree-shaking. The service is only instantiated the first time it is requested (injected) by any component or service. If the service is never used throughout the application, it won't be included in the final build, thus optimizing the application's bundle size.
  3. **Global Access:** Once instantiated by the root injector, this single instance is cached. Every subsequent request for the same service type from anywhere in the application will return this cached instance.



**Why is the Singleton Pattern Important for Services?**

  * **State Management:** Services are often used to hold and manage application state that needs to be shared across multiple components. For example, a `UserService` might store the currently logged-in user's information. If every component got its own instance, each would have a different (and likely incorrect) view of the user's state. A singleton ensures all components see the same user data.
  * **Configuration:** Application-wide configuration settings, such as API endpoints or feature flags, are ideal candidates for singleton services. This ensures that all parts of the application use the same configuration values.
  * **Shared Resources:** Services that manage shared resources, like a connection to a WebSocket or a cache for frequently accessed data, benefit greatly from the singleton pattern to prevent resource contention and ensure consistent access.
  * **Data Fetching:** When fetching data from an API (e.g., using Angular's HTTP Client), a singleton service can manage the data fetching logic. This prevents multiple components from making redundant requests for the same data, improving performance and reducing server load.
  * **Logging:** A singleton `LoggerService` can collect logs from all over the application into a single, centralized log stream.



**Example Scenario: A Singleton Configuration Service**

Consider a service that holds API base URLs:
    
    
    // src/app/services/config.service.ts
    import { Injectable } from '@angular/core';
    
    @Injectable({
      providedIn: 'root' // This makes it a singleton service
    })
    export class ConfigService {
      private apiBaseUrl: string = 'https://api.example.com/v1';
      private enableFeatureX: boolean = true;
    
      constructor() { 
        console.log('ConfigService instance created (singleton).');
      }
    
      getApiBaseUrl(): string {
        return this.apiBaseUrl;
      }
    
      isFeatureXEnabled(): boolean {
        return this.enableFeatureX;
      }
    }
    

Now, if multiple components inject `ConfigService`, they will all receive the same instance, and thus the same values for `apiBaseUrl` and `enableFeatureX`.
    
    
    // src/app/components/dashboard/dashboard.component.ts
    import { Component, OnInit } from '@angular/core';
    import { ConfigService } from '../../services/config.service';
    
    @Component({
      selector: 'app-dashboard',
      template: `
    
    API Base URL: {{ apiUrl }}
    
    Feature X Enabled: {{ featureXStatus }}
    
    `
    })
    export class DashboardComponent implements OnInit {
      apiUrl: string;
      featureXStatus: boolean;
    
      constructor(private configService: ConfigService) { }
    
      ngOnInit(): void {
        this.apiUrl = this.configService.getApiBaseUrl();
        this.featureXStatus = this.configService.isFeatureXEnabled();
      }
    }
    
    
    
    // src/app/components/settings/settings.component.ts
    import { Component, OnInit } from '@angular/core';
    import { ConfigService } from '../../services/config.service';
    
    @Component({
      selector: 'app-settings',
      template: `
    
    Current API Base URL: {{ currentApiUrl }}
    
    Feature X Status: {{ featureXStatus }}
    
    Toggle Feature X`
    })
    export class SettingsComponent implements OnInit {
      currentApiUrl: string;
      featureXStatus: boolean;
    
      constructor(private configService: ConfigService) { }
    
      ngOnInit(): void {
        this.currentApiUrl = this.configService.getApiBaseUrl();
        this.featureXStatus = this.configService.isFeatureXEnabled();
      }
    
      toggleFeatureX(): void {
        // Note: This example doesn't show how to modify the singleton state, 
        // but if ConfigService had a method to update enableFeatureX, 
        // all components would see the change.
        console.log('Toggling Feature X...');
      }
    }
    

In this example, both `DashboardComponent` and `SettingsComponent` will receive the same instance of `ConfigService`. If `ConfigService` were designed to allow modification of its state (e.g., a method to update `enableFeatureX`), any change made through one component would be immediately reflected when accessed by another component, thanks to the singleton nature of the service.

The singleton pattern, facilitated by Angular's DI and `providedIn: 'root'`, is a powerful tool for building maintainable and efficient applications by ensuring a single, consistent source of truth for shared data and logic.

## 8. Hands-On: Creating and Injecting a Mock Data Service

Now, let's put our knowledge into practice by creating a simple service to provide mock data and then injecting it into a component. This hands-on exercise will solidify your understanding of service creation, the `@Injectable()` decorator, and constructor injection.

**Objective:**

We will create a `MockDataService` that returns a predefined list of items. Then, we will create a component that injects this service and displays the items in a list.

**Step 1: Generate the Service using the CLI**

Open your terminal in the root directory of your Angular project and run the following command:
    
    
    ng generate service services/mock-data

This command will create two files:

  * `src/app/services/mock-data.service.ts`

  * `src/app/services/mock-data.service.spec.ts`




**Step 2: Implement the Mock Data Service**

Open the generated `src/app/services/mock-data.service.ts` file and modify it as follows. We'll ensure it's provided in root and add a method to return mock data.
    
    
    import { Injectable } from '@angular/core';
    
    @Injectable({
      providedIn: 'root' // Ensures this service is a singleton and available app-wide
    })
    export class MockDataService {
    
      constructor() { 
        console.log('MockDataService instance created.');
      }
    
      /**
       * Returns a predefined list of mock items.
       * @returns {string[]} An array of mock item strings.
       */
      getMockItems(): string[] {
        console.log('MockDataService: Providing mock items...');
        return [
          'Angular Fundamentals Book',
          'ASP.NET Core Development Guide',
          'Full Stack Development Course',
          'Advanced JavaScript Patterns',
          'Database Design Principles'
        ];
      }
    }
    

**Explanation:**

  * The `@Injectable({ providedIn: 'root' })` decorator makes this service a singleton and available throughout the application without needing to be explicitly declared in any module's `providers` array.

  * The `getMockItems()` method simply returns a hardcoded array of strings, simulating data that might otherwise come from an API.

  * The console log in the constructor helps us verify when the service instance is created (it should only appear once).




**Step 3: Generate a Component to Display the Data**

Now, let's generate a component that will consume our service. We'll call it `ItemList`.
    
    
    ng generate component components/item-list

This will create a folder `src/app/components/item-list` with the component's files.

**Step 4: Inject the Service into the Component**

Open the component's TypeScript file, `src/app/components/item-list/item-list.component.ts`, and modify it to inject `MockDataService` and use its method.
    
    
    import { Component, OnInit } from '@angular/core';
    import { MockDataService } from '../../services/mock-data.service'; // Adjust path as necessary
    
    @Component({
      selector: 'app-item-list',
      templateUrl: './item-list.component.html',
      styleUrls: ['./item-list.component.css']
    })
    export class ItemListComponent implements OnInit {
    
      items: string[] = [];
    
      /**
       * Constructor injects the MockDataService.
       * The 'private' keyword automatically creates a property 'mockDataService' 
       * of type MockDataService and assigns the injected instance to it.
       * @param {MockDataService} mockDataService - The injected service instance.
       */
      constructor(private mockDataService: MockDataService) { }
    
      ngOnInit(): void {
        console.log('ItemListComponent: Initializing...');
        // Call the method from the injected service
        this.items = this.mockDataService.getMockItems();
        console.log('ItemListComponent: Items loaded:', this.items);
      }
    }
    

**Explanation:**

  * We import `MockDataService`.

  * In the constructor, `private mockDataService: MockDataService` tells Angular to inject an instance of `MockDataService`.

  * The `private` keyword makes this injected instance available as `this.mockDataService` within the component.

  * In `ngOnInit`, we call `this.mockDataService.getMockItems()` to fetch the data and assign it to the component's `items` property.




**Step 5: Display the Items in the Component's Template**

Now, open the component's HTML template file, `src/app/components/item-list/item-list.component.html`, and add the following code to display the list:
    
    
    <!-- src/app/components/item-list/item-list.component.html -->
    
    <h2>Available Resources</h2>
    
    <!-- Display a loading message while the items are being fetched -->
    <div *ngIf="isLoading">Loading items...</div>
    
    <!-- Display the list of items once they are available -->
    <ul *ngIf="!isLoading">
      <li *ngFor="let item of items">{{ item }}</li>
    </ul>

**Step 6: Use the Component in Your Application**
